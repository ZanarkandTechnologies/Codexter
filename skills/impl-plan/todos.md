# Todos

- [ ] Read the active ticket first, then read the relevant PRD, specs, memory, troubles, and nearby code.
- [ ] Decide whether this is really one executable slice or whether it should be split first.
- [ ] Compare 3 viable options and recommend one clearly.
- [ ] Start the ticket plan with a short `Human` lane before the deeper `Agent` lane.
- [ ] Make the top lane carry `Decision`, diagram, `Signature Sketch`, `B -> A`, proof, and ask.
- [ ] When the work is material or ambiguous, add `User Story`, `User Pain / JTBD`, `Non-Goals`, and a concrete `High-Fidelity Example` only in the lower section when they actually reduce ambiguity.
- [ ] Make proof concrete and observable rather than generic.
- [ ] If the change is high-risk or contentious, run the `--consensus` version of this skill.
- [ ] Pitch the plan to the user for approval and revise it before build starts.
- [ ] Keep planning separate from implementation and leave the ticket in `review` until approval exists.
