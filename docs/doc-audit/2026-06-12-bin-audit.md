# Bin Directory Audit

## Question

Which `bin/` files are still relevant, which belong in `bin/`, which should move
under their owning skills, and which need stale/deletion review?

## Current Finding

`bin/` is doing too many jobs:

- live installed hook/runtime binaries
- shared Python modules imported by hook/runtime code
- repo validators and registry generators
- skill-owned command helpers
- tests for all of the above
- transitional or low-reference utilities

The largest structural problem was install scope. `install.sh` used to symlink
every top-level `bin/*` file into `~/.codex/bin`, so tests and maintenance
helpers became live installed binaries even when they were not hook/runtime
surfaces. The first cleanup pass changed install to an explicit allowlist and
moved repo-wide validators under `bin/validators/`, while leaving compatibility
wrappers at the old top-level command paths.

| `sync_frontend_pi_skills.py` | `skills/delegate-frontend/scripts/` or `skills/delegate-cli/scripts/` | Profile-specific delegate helper, not repo-wide runtime. |

## Tests

`bin/test_*.py` files are source tests, not installed binaries. Keeping them next
to their targets is fine for now, but `install.sh` should not symlink them into
`~/.codex/bin`.

Suggested install rule:

```text
installable_bin(file) -> true only for hook/runtime/operator command surfaces
```

Initial install allowlist:

- `capture_user_turn.py`
- `farplane_boards.py`
- `farplane_compute.py`
- `farplane_invocation.py`
- `notify.py`
- `runtime_telemetry.py`
- `self_improve_hook_probe.py`
- `stop_hook.py`
- `stop_hook_output.schema.json`
- `ticket-runtime`
- `ticket_runtime.py`
- `user_turn.py`

Everything else can stay in repo source and be invoked via repo-relative paths
when doing development or maintenance.

## Recommended Next Cleanup

1. Add a test or shell check that installed bin files exclude `test_*.py`,
   `__pycache__`, and repo-only validators.
2. Move skill-owned implementations one at a time, leaving small compatibility
   wrappers in `bin/` where external docs still reference the old command.
3. Re-run:

```bash
python3 bin/validators/check_doc_refs.py
python3 skills/skill-maintenance/scripts/check_skills.py --write
python3 -m unittest discover -s bin -p 'test_*.py'
```
